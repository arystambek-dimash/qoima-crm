# Дизайн: вкладка «Клиенты» + Telegram-уведомления ответственным

Дата: 2026-07-22
Статус: утверждён

## Цель

Дать админу возможность управлять аккаунтами клиентов из интерфейса CRM (создать, изменить данные, сменить пароль) и передавать эти аккаунты клиентам. Когда клиент создаёт задачу, ответственные сотрудники проекта получают личное уведомление в Telegram.

## Контекст (что уже есть)

- Роль клиента `collaborator` в `User` (`backend/src/users/models.py`, `core/enums.py`). Клиенты логинятся и видят «Кабинет клиента» (Главная, Проекты, Профиль).
- Привязка клиента к проекту: `Deal.user` (основной) и `Deal.collaborators` (M2M) — управляется из формы проекта (`frontend/src/app/(app)/deals/deal-form-dialog.tsx`).
- Ответственные сотрудники проекта: `Deal.responsibles` (M2M, роль `employee`).
- Задача от клиента уже создаётся со статусом согласования `pending` и уходит в общий Telegram-чат согласования с кнопками одобрить/отклонить (`backend/src/onboards/views/task_views.py`, `backend/src/telegram_bot/services/task_approval.py`).
- Инфраструктура Telegram: `TelegramClient.send_message(chat_id, text)` (`backend/src/telegram_bot/services/telegram.py`), у `User` есть `telegram_id`.

## Решения (утверждены)

1. **Уведомления:** личные сообщения ответственным проекта **плюс** существующий чат согласования (ничего не ломаем).
2. **Пароль:** админ вводит пароль сам — при создании клиента и при смене пароля.
3. **Доступ к вкладке:** только суперпользователь.
4. **Бэкенд:** без новой модели — клиент это `User` с ролью `collaborator`; эндпоинты поверх `User`.

## 1. Бэкенд API

Новый `ClientViewSet` в приложении `users`, маршрут `/api/clients/`, разрешение — только суперпользователь (403 для сотрудников и клиентов):

- `GET /api/clients/` — список пользователей с ролью `collaborator`: id, имя, фамилия, email, проекты, в которых участвует (по `primary_deals` + `collaborator_deals`).
- `POST /api/clients/` — создание: `first_name`, `last_name`, `email`, `password`. `username` выводится из email, `role` всегда `collaborator` (не принимается от клиента API).
- `PATCH /api/clients/{id}/` — изменение `first_name`, `last_name`, `email`.
- `POST /api/clients/{id}/set-password/` — тело `{password}`, `set_password()` + сохранение.

Требование к паролю (создание и смена): минимум 8 символов, проверяется на бэкенде сериализатором и дублируется подсказкой в форме.

Queryset жёстко ограничен `role=collaborator` — через эти эндпоинты нельзя увидеть или изменить сотрудника.

## 2. Telegram-уведомление ответственным

Новый сервис в `backend/src/telegram_bot/services/` (например, `task_notifications.py`):

- Функция принимает id задачи, находит проект через `task.category.onboard.deal`, берёт `deal.responsibles`, и каждому с непустым `telegram_id` шлёт личное сообщение через `TelegramClient.send_message`.
- Вызов — из `TaskViewSet.create` рядом с существующей отправкой в чат согласования, только для задач, созданных клиентом (`collaborator`), через `transaction.on_commit`.

Текст сообщения:

```
🆕 Новая задача от клиента <Имя Фамилия>
Проект: <название проекта>
Задача: <название> (срок: <date_end>)
Статус: на согласовании
```

Чат согласования и кнопки одобрить/отклонить работают без изменений.

## 3. Фронтенд

- В `NavItem` (`frontend/src/components/app-shell/nav.tsx`) добавляется флаг `superuserOnly?: boolean`; `useNavVisibility` (`frontend/src/lib/permissions.ts`) скрывает такие пункты для всех, кроме суперпользователя.
- Новый пункт «Клиенты» в группе «Компания» (иконка из lucide, например `Contact`), маршрут `/clients`.
- Страница `frontend/src/app/(app)/clients/page.tsx` по образцу «Сотрудников»:
  - таблица: имя, email, проекты;
  - кнопка «Добавить клиента» — диалог с полями имя, фамилия, email, пароль;
  - в строке: «Редактировать» (имя, фамилия, email) и «Сменить пароль» (одно поле пароля).
- Эндпоинты добавляются в `frontend/src/lib/endpoints.ts` по существующему образцу.

## 4. Ошибки и крайние случаи

- У проекта нет ответственных, или у ответственного не привязан Telegram → уведомление пропускается (лог-запись), задача создаётся нормально.
- Сбой Telegram API не ломает создание задачи: отправка после коммита транзакции, исключения перехватываются и логируются.
- Email уже занят → ошибка валидации сериализатора, показывается в форме.
- Задача клиента не привязана к проекту (нет цепочки category→onboard→deal) → уведомление пропускается.

## 5. Тесты

Бэкенд (по существующим паттернам тестов проекта):

- Доступ: суперпользователь получает список/создаёт/меняет; сотрудник и клиент получают 403.
- Создание клиента: роль `collaborator`, вход по email/паролю работает.
- Смена пароля: старый перестаёт работать, новый работает.
- Уведомления: при создании задачи клиентом — сообщение каждому ответственному с `telegram_id` (Telegram-клиент замокан); без `telegram_id` — пропуск; сбой отправки не ломает создание задачи.

## Вне рамок

- Удаление/деактивация клиентов (можно добавить позже).
- Дополнительные поля клиента (компания, телефон) — потребуют отдельной модели.
- Уведомления клиентам о статусе их задач.
